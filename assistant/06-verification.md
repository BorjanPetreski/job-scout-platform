# Verification (re-verify before every draft)

The scanner checked liveness when it found the role, but time passes before the user works the
queue — postings die, hours change, a range appears. **Re-verify fetch-first, every time, before
drafting.** Never draft against an unverified posting.

## Fetch-first

Fetch the posting URL and re-read the hard facts, checking each against the profile snapshot:

- **Still live?** A 404 / 410 / "position filled" / expired marker → dead.
- **Eligibility** — is the location / remote-region / visa situation a fit for this profile
  (from the snapshot's eligibility + location)?
- **Hours** — part-time vs full-time, against what the profile accepts.
- **Salary** — against the snapshot's floor (or judgment estimation if the profile is floorless).
- **Employment type** — B2B / contract / full-time (drives the closing gate).

## On a wall

Many good postings sit behind a JS shell, a login, or a thin card (LinkedIn especially). When
you can't read the posting directly, **ask the user to paste the JD or confirm the key facts.**
Don't guess, and don't draft on the card alone. An honest "I couldn't read this one — can you
paste it?" is the correct move.

## On a dead posting

Set the Passed/Seen row to `Stale/Expired` with a dated note (see the write contract) and move to
the next role. Don't draft.

## Why this is mandatory

A drafted application against a stale or mis-read posting wastes the user's time and can embarrass
them (applying to a filled role, quoting the wrong hours). Verification is cheap; a bad draft is
not. It is never optional.
